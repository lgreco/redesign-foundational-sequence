## Array Resize Strategies (June 2)

We explored the trade-off at the heart of dynamic arrays: when the underlying array fills up, we must allocate a new one and copy all existing elements — a cost proportional to the current size. Adding just one extra slot minimizes wasted memory but forces copying on every add. Doubling wastes up to half the space but copies rarely. We explored percentage-based resizing (e.g., 10%) with a ceiling function to guarantee at least one new slot, balancing both concerns.

To give users control, we evolved the `DynamicArray` class to accept optional `capacity` and `resize_by` arguments with class-level defaults (`DEFAULT_CAPACITY = 4`, `DEFAULT_RESIZE_BY = 2`). We demonstrated the object with resize factors of 2, 4, and 0.1 — and deliberately ended class by revealing a crash at `resize_by=0.1`, leaving the diagnosis for the next day.

## Debugging and Refactoring (June 3)

With help from Alexander, Temeeka, and Dutch, we traced the bug: multiplying an integer capacity by a float resize factor produces a float, and Python's `int()` floors the result — so a capacity of 3 grows by 10% to 3.3, which `int()` rounds down to 3, the same size. The fix: `math.ceil()`, which rounds up. We compared two import styles — `import math` then `math.ceil(...)`, versus `from math import ceil` then `ceil(...)` — and noted both are valid.

## Naming, Privacy, and Encapsulation

We generalized the internal container name from `zip_codes` to `_underlying` — generic enough for any use. Single leading underscores on `_underlying`, `_capacity`, `_size`, and `_resize` signal "please don't access this directly." Python relies on professional courtesy here rather than enforcement; double underscores add name mangling for extra friction. The analogy: a car's dashboard (public interface) vs. its engine internals (private). The weekend assignment includes writing a method that safely exposes the object's size without violating encapsulation.
