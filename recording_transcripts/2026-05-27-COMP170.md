## Revisiting the Interest Program

We returned to the compound interest program from yesterday, using it as a springboard for one of the most important ideas in programming: **separation of concerns** — keeping input, logic, and output in distinct, named pieces of code rather than mixing them together in one block.

## Functions: A Brief History

We discussed how a named chunk of code that handles one task has gone by many names over the decades: *procedure* (1950s), *subroutine* (Fortran, 1960s), and today most commonly *function* or *method*. The coffee shop analogy: ordering a caramel macchiato — you name the drink and trust the barista to handle the steps. You don't recite the recipe every time. 

## Rewriting the Program with Three Functions

We wrote `interest_pro.py` side-by-side with the original one-block version. The new program has three functions:

- `compute_future_value(principal, rate, term)` — the math: `principal * (1 + rate) ** term`, returns the total.
- `read_inputs()` — asks the user for all three values and returns them.
- `show_result(principal, rate, term, total)` — prints the inputs and the result.

The main program collapses to three lines: call `read_inputs`, call `compute_future_value`, call `show_result`. We caught two typos live (a mismatched variable name and `show_result` vs. `show_results`) and then saw how an IDE catches those errors with a red underline before you even run the code.

## Comments and Indentation

We closed with two habits worth building now. First, write comments to explain *why* the code does what it does — especially useful months later when the details are fuzzy. Second, consistent indentation matters in Python: it defines **scope**, meaning which lines belong inside a function. With Bella's question, we confirmed that comments don't technically have to follow indentation rules, but should — inconsistent spacing makes code harder to read and maintain.
