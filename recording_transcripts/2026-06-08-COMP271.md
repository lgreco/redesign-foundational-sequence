## Assignment Review: `__len__` and `get_size`

We reviewed the assignment on the `DynamicArray` class, focusing on two methods that do the same thing: `__len__` and `get_size`. Both return `self._size` — the number of elements currently stored, not the underlying array's total capacity. We compared the hotel analogy: capacity is how many rooms exist; size is how many guests are checked in. We also explored implementing `get_size` as a call to `len(self)` so that any future change to the internal variable name only needs to be made in one place.

## Getter Methods and Private Variables

We discussed *getter methods* (also called *accessors*): methods named `get_<attribute>` that give outside code controlled access to private data. Private variables (marked with a leading underscore, e.g., `_size`) should not be read directly from outside the class. The analogy: asking someone their birthday is fine; grabbing their wallet to check the date is not.

## The `self` Keyword

Python accepts any name for the object reference parameter — we demonstrated `carrots`, `broccoli`, `cabbage`, and `corn`, one per student's least-favorite vegetable. All work correctly. By convention, however, we use `self` for consistency and readability. Java enforces the keyword `this` instead.

## Bounds Checking in `get`

The `get(index)` method needs a guard to prevent out-of-bounds access. Without checking that `index` falls in `[0, self._size)`, a caller can request a position that doesn't exist and crash the program.

## Searching: `index_of`

We implemented `index_of` two ways — a `for` loop and a `while` loop — and compared them. The `for`-loop version has two `return` statements and a subtle bug: if the value isn't found, it returns the last loop index rather than −1. The `while` loop is preferred for searches because its condition can encode both "not yet found" and "not yet exhausted all positions," making the logic explicit and avoiding the need for `break`.

## Structured Programming: Single Entry / Single Exit

We introduced Dijkstra's principle: methods should have one entry point (the header) and one exit point (one `return` statement). For short methods that fit on a screen, multiple `return` statements are a style problem. The `break` statement is essentially a disguised `goto` — it violates structured programming and should be avoided in favor of expressing the termination condition in the loop's header.
