# Analysis of COMP 141, 163, and 170

The topics currently covered in COMP 141, 163, and 170 are listed below. A gap analysis is provided after the listings, suggesting topics that may be deferred for later in the curriculum.

## Current coverage

### COMP 141 — Introduction to Computing Tools and Techniques

  * Environment setup: MacOS Terminal, Windows Subsystem for Linux, native Ubuntu Linux
  * Command-line operation: calling programs, passing options and arguments
  * Navigation: moving through the filesystem, exploring directory structure
  * File and directory manipulation: `cp`, `mv`, `rm`, `mkdir`, `cd`, `ln`, `cat`
  * Filesystem concepts: files, directories, hard and symbolic links
  * User and group permissions: `chmod`, `chown`
  * Working with commands: command type, help systems (`man`, `--help`)
  * I/O redirection: stdin, stdout, stderr, redirecting with `>`, `>>`, `<`
  * Pipelines: chaining commands with `|`
  * Shell expansion: globbing, brace expansion, tilde expansion, parameter and arithmetic expansion
  * Stream commands: `cat`, `grep`, `head`, `tail`, `tee`
  * Advanced keyboard tricks: readline shortcuts, command history, tab completion
  * Regular expressions: syntax, usage in `grep` and other tools
  * Processes: running and monitoring programs, foreground and background control, job control (`jobs`, `fg`, `bg`, `kill`)
  * Shell environment: reading and setting variables, significant system variables (`PATH`, `USER`, `HOME`)
  * Prompt customization: modifying `PS1`
  * Text editing: fast and advanced operation with Vi/Vim (modes, navigation, editing, search/replace)
  * Package management: software installation, removal, and update (`apt`, `brew`)
  * Searching for files: `find`, `locate`
  * Archiving and file compression: `tar`, `zip`, `gzip`, `bzip2`
  * Text processing: `sort`, `uniq`, `cut`, `paste`, `join`, `diff`, `sed`
  * Networking: remote access (`ssh`), file transfer (`scp`, `sftp`), network configuration inspection
  * Terminal multiplexing: managing multiple terminal sessions
  * Program compilation: compiling C and Java programs from the command line
  * Interpreted languages: running Python scripts from the shell
  * Shell scripting: writing and executing scripts, shebang line, comments
  * Shell scripting — keyboard input: reading user input with `read`
  * Shell scripting — positional parameters: `$1`, `$2`, `$@`, `$#`
  * Shell scripting — branching: `if`/`elif`/`else` conditionals, test expressions
  * Shell scripting — looping: `for` loops, `while` loops, looping over files
  * Version control: Git profile creation, `init`, `add`, `commit`, `status`, `log`, `push`, `pull`

### COMP 163 — Discrete Structures

  * Set theory: set notation, membership, subsets, power sets
  * Set operations: union, intersection, difference, complement, Cartesian product
  * Propositional logic: propositions, truth values, logical connectives (`∧`, `∨`, `¬`, `→`, `↔`)
  * Truth tables: constructing and interpreting truth tables
  * Logical equivalences: De Morgan's laws, distributive laws, other equivalences
  * Quantifiers: universal (`∀`) and existential (`∃`) quantifiers, negation of quantified statements
  * Logical inference rules: modus ponens, modus tollens, hypothetical syllogism, disjunctive syllogism
  * Boolean algebra: axioms, simplification, duality
  * Combinatorial circuits: logic gates (AND, OR, NOT), circuit design and analysis
  * Methods of proof: direct proof, proof by contrapositive, proof by contradiction
  * Mathematical induction: weak induction, strong induction
  * Relations: definition, properties (reflexive, symmetric, transitive, antisymmetric), equivalence relations, partial orders
  * Functions: definition, domain and codomain, injective, surjective, bijective functions, composition, inverse
  * Combinatorics: counting principles (addition rule, multiplication rule)
  * Permutations and combinations: `P(n,r)`, `C(n,r)`, binomial theorem
  * Pigeonhole principle
  * Number systems: binary, hexadecimal, and floating-point representation and arithmetic
  * Probability theory: sample spaces, events, basic probability axioms, conditional probability
  * Algorithm time complexity analysis: Big-O notation, best/worst/average case, comparing growth rates

### COMP 170 — Introduction to Object-Oriented Programming (Python)

  * Introduction to Python programming: running a program, `print()`, program structure, syntax errors
  * What is an algorithm: definition, properties, step-by-step problem decomposition
  * Defining and calling functions: function syntax, the call stack, built-in vs. user-defined functions
  * Basic data concepts: data types (int, float, str, bool), literals, type conversion
  * Variables: declaration, assignment, naming conventions, scope
  * Expressions and operators: arithmetic operators, operator precedence, integer vs. float division
  * The `for` loop: iterating over a range, loop variable, loop body
  * `range()`: start, stop, step arguments; counting and accumulating with loops
  * Problem solving with loops: loop patterns, accumulator pattern, tracing execution
  * Function parameters: positional parameters, default values, passing arguments
  * Return values: `return` statement, returning multiple values, functions as expressions
  * The `math` module: common math functions (`sqrt`, `floor`, `ceil`, `log`, etc.), importing modules
  * Interactive programs: `input()`, converting user input, prompting patterns
  * The `random` module: `random()`, `randint()`, seeding, simulating randomness
  * The `if` statement: `if`, `elif`, `else`; Boolean conditions; nested conditionals
  * Cumulative algorithms: running totals, running max/min, building results incrementally
  * Introduction to lists: list literals, indexing, `len()`, iterating over a list with `for`
  * Text processing: string methods (`split`, `strip`, `lower`, `upper`, `replace`, `find`), string iteration
  * Methods with conditional execution: calling methods inside conditionals, chaining method calls
  * Exceptions: what exceptions are, common exception types (`ValueError`, `TypeError`, `IndexError`)
  * Pre- and post-conditions: specifying what a function expects and guarantees
  * Indefinite loops: `while` loops, loop conditions, infinite loops, `break`
  * Program logic and Boolean expressions: compound conditions with `and`, `or`, `not`; short-circuit evaluation
  * Relational expressions: comparing values, chained comparisons
  * Assertions: `assert` statement, using assertions to check invariants
  * `try`/`except`: catching exceptions, handling errors gracefully, `finally`
  * Introduction to text file I/O: opening files with `open()`, reading lines, closing files, `with` statement
  * More file processing: writing to files, appending, iterating over file contents, reading structured data
  * More about lists: list methods (`append`, `insert`, `remove`, `sort`, `pop`), list slicing, nested lists
  * List algorithms: searching, counting, filtering elements; list comprehension basics
  * Object-oriented programming: classes and objects, object state (instance variables) and behavior (methods)
  * Constructors: `__init__` method, initializing object state, creating instances
  * Calling methods on objects: dot notation, `self` parameter, method vs. function

## Gap Analysis: Additions and Deferrals

This section identifies topics missing from the three source courses that are needed for success in
COMP 271 (Data Structures I) and COMP 272 (Data Structures II), and topics currently present
that are better deferred to later courses. Analysis is based on the stated learning outcomes and
prerequisites of COMP 271 and COMP 272.

**Items proposed for removal/deferral** are not meant to be removed completely from an undergraduate CS curriculum. Rather, we propose to introduce them in later courses and in closer proximity with more relevant subject matter.

### Topics to add

**COMP 170**

- **Inheritance and polymorphism.** COMP 271 explicitly names these as OOP skills it builds on. The current COMP 170 schedule ends at a single class with a constructor; students arrive at COMP 271 with no exposure to subclasses, method overriding, or `isinstance`. This is the most critical gap.
- **Recursion.** Absent from the Spring 2025 schedule but essential for understanding trees, graph traversal, and algorithm analysis in COMP 271 and 272. Needs to appear somewhere in the integrated sequence.
- **Python dictionaries.** COMP 271 covers hash tables; students who have never used a Python `dict` will struggle from week one. Basic creation, lookup, iteration, and use cases are not covered in the BPP chapters currently scheduled (Ch 1–7, 11).
- **Mutability and object references.** The distinction between mutable and immutable types, and the fact that assignment copies a reference rather than a value, is prerequisite for reasoning about linked structures. COMP 170 mentions string immutability in passing but never treats the concept systematically.
- **Systematic testing.** COMP 271 lists "testing and documenting code" as a learning outcome. COMP 170 covers debugging but not test writing — not even simple `assert`-based checks or the habit of writing a `test_` function alongside a function under development.
- **Docstrings.** COMP 271 expects documented code. COMP 170 mentions comments but not docstrings, the Python convention for function and class documentation. One session on `"""..."""`, what goes in a docstring, and how `help()` uses it would suffice.
- **Designing with multiple interacting classes.** COMP 170 ends with a single class. Students need at least one exercise where two or three classes interact (e.g., a `Node` used by a `LinkedList`) before encountering that pattern at speed in COMP 271.

**COMP 163**

- **Recurrence relations as a formal topic.** The schedule covers proof by induction, which touches recurrences implicitly, but never names them. COMP 271 and 272 use recurrence relations to analyze recursive algorithms (e.g., merge sort: T(n) = 2T(n/2) + n). A brief, explicit treatment — setting up and solving simple recurrences — would meaningfully help students follow algorithm analysis in the downstream courses.

**COMP 141**

- **Makefiles and build automation.** Whether a student's first COMP 271 submission contains a `Makefile` is identified as a binary indicator of CLI carry-forward. `make` is implied by "compiling programs" in the COMP 141 topic list but never explicitly taught. Even a simple two-target Makefile (build and clean) establishes the habit.

### Topics to defer

**COMP 141**

- **Networking commands** (`ssh`, `scp`, `sftp`, network configuration inspection). Valuable in systems and DevOps contexts but low direct-prerequisite value for COMP 271 and 272. Better placed in a systems, networking, or distributed systems course. Reducing this frees time for deeper Git coverage and shell scripting.
- **C program compilation.** The students' onward path runs through Python (COMP 271) and then Java (COMP 272). C is not on that road. This could be reduced to a brief demo rather than a topic in its own right.

**COMP 163**

- **Combinatorial circuits and logic gates** (Boolean algebra Ch 11, two full class sessions in the schedule). This material belongs in a digital logic or computer architecture course. It has essentially no direct application in COMP 271 or 272 and consumes sessions that could go toward recurrence relations or introductory graph theory.

**COMP 170**

- **Depth of file I/O coverage.** Four weeks are spent on text file processing (Ch 6 preview in week 10, full treatment in week 12, plus Ch 6.1 revisited). A lighter pass — `open()`, reading line by line, writing, the `with` statement — is sufficient preparation for COMP 271. The depth of structured file parsing is more appropriate to a data-focused or systems course.

## COMP 163 Topic-by-Topic Review

The COMP 163 syllabus follows Johnsonbaugh's *Discrete Mathematics*, a comprehensive reference text. Not every topic it covers is equally justified for a curriculum whose next stops are COMP 271 (data structures in Python) and COMP 272 (data structures in Java). The following reviews each topic against that specific downstream context.

### Topics that clearly earn their place

- **Set theory and set operations.** Sets, maps, and related ADTs are the subject matter of COMP 272. The vocabulary and operations (union, intersection, complement, Cartesian product) are used constantly. Keep in full.

- **Propositional logic, truth tables, and De Morgan's laws.** Boolean conditions are the connective tissue of programming. De Morgan's laws in particular are used whenever a programmer inverts a compound condition. The formal treatment is justified. The deeper logical equivalences beyond De Morgan's can be lightened.

- **Mathematical induction — the intuition.** The base-case/inductive-step pattern is the formal backbone of reasoning about recursive algorithms and recursive data structures. When arguing that a property holds for every node in a tree, or that a recursive function always terminates with the right answer, you are implicitly using structural induction. That reasoning pattern appears throughout COMP 271 and 272, and one session grounding it — with an explicit connection to why recursion works — is well justified.

- **Direct proof and proof by contradiction.** These develop the mathematical maturity that COMP 271 and 272 expect when students read algorithm correctness arguments. Worth keeping, even if students never write formal proofs again — the discipline of "assume X, show Y follows" transfers to debugging and testing reasoning.

- **Relations.** Equivalence relations underlie the bucket structure of hash tables; partial orders underlie comparison-based sorting and tree ordering. Both appear explicitly in COMP 271 and 272. Keep.

- **Functions (domain, codomain, range).** Hash functions, tree node mappings, and API design all rest on the concept of a function as a well-defined input-output relationship. Keep the core definition and properties.

- **Basic counting: addition and multiplication rules, permutations, combinations.** Used in algorithm analysis (counting operations, bounding search spaces). Keep, with the binomial theorem as a candidate for reduction (see below).

- **Binary and hexadecimal number systems.** Practically useful for every programming course, especially when students encounter memory addresses, bitmasks, or integer overflow. Keep.

- **Algorithm time complexity and Big-O notation.** The most directly prerequisite topic for COMP 271 and 272, both of which open with complexity analysis. Keep in full.

- **Basic probability: sample spaces, events, probability axioms.** Useful for expected-case analysis (average-case behavior of hash tables, quicksort). A basic treatment is justified.

### Topics that need a second look

**We are not proposing** to eliminate these topics from the curriculum. Instead we propose to defer them for later courses where and when they'll be more relevant.

- **Logical inference rules (modus ponens, modus tollens, hypothetical syllogism, disjunctive syllogism).** These named schemas are the machinery of formal proof theory and appear in automated theorem proving and formal verification — neither of which is in the *immediate* vicinity of this coursework. The reasoning habits they represent are genuinely useful, but students absorb them naturally through direct proof and induction without needing to memorize Latin names. One brief session connecting them to "how we reason in proofs" may be useful. Further review may be incorporated in introductory material for more relevant/advanced courses.

- **Quantifiers (∀, ∃) and negation of quantified statements.** The notation is useful — pre/post conditions and loop invariants are naturally expressed with quantifiers — but the formal manipulation of negated quantified statements ("there exists no x such that...") is more relevant to formal methods than to data structures. **Recommend: keep the notation and the basic reading of quantified statements; drop the formal negation manipulation.**

- **Boolean algebra (axioms, simplification, duality).** The main downstream use of Boolean algebra in this curriculum is simplifying boolean expressions in code, which De Morgan's laws (already in the logic section) already cover. The algebraic system in full — with duality, axiomatic development, and Huntington's postulates — is primarily motivated by circuit design. If combinatorial circuits are deferred (as recommended earlier), Boolean algebra loses its main justification. **Recommend: fold De Morgan's into the propositional logic unit and drop the standalone Boolean algebra treatment.**

- **Combinatorial circuits and logic gates.** Already flagged for deferral to a digital logic or computer architecture course. Repeating it here for emphasis: nothing in COMP 271 or 272 requires knowing how to design or analyze a combinatorial circuit. **Recommend: defer entirely.**

- **Mathematical induction — the formalism.** Students in COMP 271 and 272 are not asked to write induction proofs. Algorithm complexity is handled via the Master Theorem (used as a recipe, not derived) and by unrolling recurrences. The second session on strong induction, and the full algebraic proof-writing exercises, go beyond what the downstream courses require. The same principle that applies to logical inference rules applies here: the reasoning habit is worth building; the formal apparatus beyond what gets reused is overhead. **Recommend: one session on the intuition and its connection to recursion; drop the second session and reduce proof-writing exercises.**

- **Proof by contrapositive.** Less commonly used than direct proof and contradiction, and the technique is partially captured by proof by contradiction anyway. It can be mentioned as a named strategy in one sentence without dedicating class time to it. **Recommend: brief mention only.**

- **Bijective functions, function composition, and inverse functions.** The practical need in this curriculum is: understand what a function is (one output per input, defined domain) and recognize when two things are "in correspondence." The formal type-theoretic apparatus of injection, surjection, bijection, composition, and inverse exceeds that need. Hash functions are not bijections; tree traversals are not function compositions. **Recommend: keep domain/codomain/range and the concept of well-definedness; reduce the formal taxonomy.**

- **The binomial theorem.** Appears in the combinatorics unit. Its direct applications in COMP 271/272 are essentially zero. It arises in probability and in analysis of certain algorithms (binomial heaps in COMP 272, arguably), but not at a level where students need to know the theorem. **Recommend: drop.**

- **Pigeonhole principle.** A one-session topic with a direct connection to hash table collisions — if there are more keys than buckets, at least two keys must share a bucket. Worth a brief treatment in that context rather than as a standalone principle. **Recommend: keep as a one-paragraph note within the hash/counting unit, not a full session.**

- **Conditional probability.** Basic probability (sample spaces, events, axioms) is useful for expected-case algorithm analysis. Conditional probability in formal detail — Bayes' theorem, conditional independence — is more relevant to a probability or machine learning course. **Recommend: keep basic probability; drop or heavily abbreviate conditional probability.**

- **Floating-point representation and arithmetic.** Binary and hexadecimal are clearly useful in programming. Floating-point representation (IEEE 754, mantissa, exponent, rounding) is more relevant to numerical methods and systems courses. Students in COMP 271/272 encounter floats as a data type, not as a representation to analyze. **Recommend: keep binary and hexadecimal; reduce floating-point to a brief awareness note.**

## Appendix

### A note on induction

The strongest argument for induction is that it is the formal backbone of reasoning about recursive algorithms and recursive data structures. When we argue that a property holds for all  
nodes in a tree, or that a recursive function terminates and produces the right answer, we are implicitly using structural induction — base case (leaf node or empty list), inductive step (if it holds for children, it holds for the parent). That reasoning pattern appears in COMP 271 and 272.

But do students in COMP 271/272 actually write induction proofs? Almost certainly not. They are asked to implement and analyze data structures, not to formally prove their correctness. A student can successfully complete both courses without ever writing a line of formal induction.

Our recommendation is to keep the intuition and reduce the formalism. One session establishing the base-case/inductive-step pattern and connecting it explicitly to why recursion works is well justified. The second session on strong induction, and the full algebraic treatment, is probably not. COMP 163 currently spends two class sessions on induction (weeks 7 and 9 in the  schedule). One would be enough if the focus shifts from proof-writing exercise to conceptual grounding for recursion.     

This is the same principle we applied to logical inference rules in the gap analysis above: the habit of reasoning is worth building, but the formal apparatus beyond what's reused downstream is overhead.
