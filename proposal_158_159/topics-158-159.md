# COMP 158 and COMP 159 ‚Äî Integrated Introductory Sequence

Each course runs 14 weeks. Class sessions are either three 50-minute meetings or two 75-minute meetings per week. All programming is done at the command line using plain text editors (Vim or nano) and Makefiles. Git is the submission mechanism from week 2 forward.

The sequence is organized around **programming concepts**. Mathematical ideas and CLI tools are introduced when they illuminate or enable the programming concept at hand ‚Äî not on a separate schedule. The flow in COMP 158 is roughly: environment ‚Üí data and types ‚Üí logic and conditionals ‚Üí functions ‚Üí sequences and vectors ‚Üí iteration ‚Üí more on lists ‚Üí sets ‚Üí number representation ‚Üí objects and classes. COMP 159 picks up with recursion and builds toward algorithm analysis, advanced OOP, hashing, and abstract data types.

## COMP 158

### Week 1 ‚Äî The Environment

Programming happens somewhere. Before writing code, students need to know where they are and how to get around. The first week establishes the terminal as the place where everything will happen: creating files, running programs, submitting work.

Students learn filesystem navigation (`ls`, `cd`, `pwd`, `mkdir`, `cp`, `mv`, `rm`), how to open and edit a file in Vim (modes, basic navigation, saving), and how to run a Python script with `python3`. The only Python this week: `print()` and comments. The point is not to learn Python yet ‚Äî it is to learn that a program is a text file you run from the shell.

Git is introduced at the end of the week: `init`, `add`, `commit`, `status`. Every piece of work from week 2 onward is submitted via `git push`.

---

### Week 2 ‚Äî Values, Types, and Variables

Programs operate on data. This week establishes what data is: values belong to types, and types determine what operations are legal.

**Data types:** `int`, `float`, `str`, `bool`. Literals. Type conversion: `int()`, `float()`, `str()`. Arithmetic operators and precedence. Variables as names for values; assignment as binding a name to a value.

**Mathematical framing:** a type is a set. `int` is (a finite subset of) ‚Ñ§; `float` approximates ‚Ñù; `str` is sequences of characters from a finite alphabet. An operation is defined on certain types and not others ‚Äî a type error is what happens when you try to apply an operation outside its domain. This framing will resurface throughout the course.

By the end of the week, students can write a script that reads a number from the user with `input()`, does arithmetic on it, and prints the result.

---

### Week 3 ‚Äî Logic and Conditionals

How does a program decide what to do? The answer is logic. Before writing `if` statements, students need to understand what a proposition is, what it means for one to be true, and how combining propositions with "and," "or," and "not" works.

**Mathematical concepts:** propositions and truth values. Connectives: ‚àß (and), ‚à® (or), ¬¨ (not). Truth tables: constructing and reading them. Logical equivalences ‚Äî in particular, De Morgan's laws:

> ¬¨(P ‚àß Q) ‚â° ¬¨P ‚à® ¬¨Q  
> ¬¨(P ‚à® Q) ‚â° ¬¨P ‚àß ¬¨Q

De Morgan's is not an abstract curiosity: it is exactly what a programmer does when negating a compound condition. Students see it in code the same session they see it in a truth table.

**Programming:** `if`, `elif`, `else`. Boolean expressions. Compound conditions with `and`, `or`, `not`. Short-circuit evaluation. Nested conditionals. The connection is explicit: a Python boolean expression is a proposition; its truth value determines which branch runs.

**CLI note:** I/O redirection (`>`, `>>`, `<`) is introduced briefly so students can redirect their program's output to a file and inspect it.

---

### Week 4 ‚Äî Functions: The Mathematical Idea

A function is a rule that maps each element of a domain to exactly one element of a codomain. This is one of the central ideas of mathematics ‚Äî and of programming. Understanding the mathematical concept first makes the programming concept sharper.

**Mathematical concepts:** function as a mapping (domain ‚Üí codomain). The one-output requirement: for each input, exactly one output. Domain, codomain, and range (the subset of the codomain actually produced). Writing functions in mathematical notation: f: A ‚Üí B, f(x) = x¬≤. Function composition: (f ‚àò g)(x) = f(g(x)).

**Programming:** `def`, parameters, `return`. The call stack. A Python function is an implementation of a mathematical function ‚Äî same structure, concrete syntax. Pure functions (output depends only on input, no side effects) vs. functions that also print, write files, or modify state. The distinction matters: pure functions are easier to reason about and test.

**CLI:** A simple Makefile with a `run` target. `make run` replaces retyping `python3 script.py`.

---

### Week 5 ‚Äî Designing and Testing Functions

Writing a function is syntactically simple. Designing one well takes discipline. This week focuses on what a function should do, how to specify it clearly, and how to verify that it does what you claim.

**Pre- and post-conditions:** a function's contract with its callers. The precondition states what the function expects (the set of valid inputs); the post-condition states what the function guarantees (the property of the output). Expressed informally at first: "given a non-empty list of numbers, returns the largest one."

**Docstrings:** the Python convention for stating pre/post conditions in code. `"""..."""` inside a function definition. What `help()` shows. One session, one rule: every function gets a docstring before anyone writes the body.

**Testing:** `assert` as a tool for checking that a function does what its docstring claims. Writing a `test_` function for each function under development. The `make test` target runs all tests before a commit. The habit being established: write the docstring, write a test, then write the code.

### Week 6 ‚Äî Sequences and Vectors

Mathematics has a precise vocabulary for ordered collections of values: sequences, vectors, arrays. Programming has lists. This week connects them, using the dot product as the central worked example.

**Mathematical concepts:** sequences as indexed families (a‚ÇÅ, a‚ÇÇ, ..., a‚Çô). Vectors as elements of ‚Ñù‚Åø ‚Äî an n-tuple of real numbers. Indexing notation: a·µ¢ refers to the i-th element. Summation notation: ‚àë·µ¢‚Çå‚ÇÅ‚Åø a·µ¢ is the sum of a‚ÇÅ through a‚Çô. The dot product of two vectors **x** and **y**: **x**¬∑**y** = ‚àë x·µ¢y·µ¢.

**Programming:** Python lists. Literals (`[1, 2, 3]`), indexing (`lst[0]`), `len()`, `append()`, slicing (`lst[1:3]`). A list is the implementation of a mathematical sequence: ordered, indexed from 0, mutable. Iteration over a list is introduced here but not yet formalized ‚Äî students see `for x in lst` as a way to visit each element.

The dot product is implemented as a loop this week, setting up the formal treatment of iteration next week.

### Week 7 ‚Äî Iteration: Operationalizing ‚àÄ and ‚àë

The `for` loop is how you say "for every element in this collection, do something." That is a direct programming realization of the universal quantifier ‚àÄ. The accumulator pattern computes ‚àë. This week makes the connection explicit.

**Mathematical concepts:** the universal quantifier ‚àÄ: "for all i in some set, P(i) holds." The `for` loop operationalizes this ‚Äî it visits every element and applies the loop body. Summation as a specific instance: ‚àë·µ¢‚Çå‚ÇÅ‚Åø f(i) is implemented as an accumulator loop. The dot product from week 6 becomes the worked example: **x**¬∑**y** = ‚àë x·µ¢y·µ¢ is translated line-by-line into a `for` loop with an accumulator.

**Programming:** `for` loop, `range()` (start, stop, step). The accumulator pattern: running total, running maximum, running minimum. Tracing loop execution by hand. `for` loops over lists and over `range`.

**CLI note:** The shell `for` loop (`for f in *.py; do python3 "$f"; done`) is shown as the same concept in a different language ‚Äî same structure, different syntax.

### Week 8 ‚Äî Searching and ‚àÉ

Not every loop runs to completion. Sometimes you stop when you find what you need. The `while` loop and `break` are how you say "keep going until you find something that satisfies this condition" ‚Äî which is the existential quantifier ‚àÉ.

**Mathematical concepts:** the existential quantifier ‚àÉ: "there exists an element in the set such that P holds." A search that returns "not found" is in effect verifying ‚àÄ¬¨P ‚Äî no element satisfies the property. Negation of quantifiers: ¬¨(‚àÄx, P(x)) ‚â° ‚àÉx, ¬¨P(x), and vice versa. Students see this in the logic of a search that either finds a match or confirms there is none.

**Programming:** `while` loop, loop conditions, `break`. Linear search: iterate until found or exhausted. `input()` in an interaction loop: keep prompting until valid input is given. The `in` operator as a built-in linear search.

**Exception handling ‚Äî first encounter:** Students have already seen `int(input())` raise a `ValueError` when the user types something that is not a number. Leaving them without vocabulary for that situation for a full semester is not a good idea. The interaction loop is the natural place to introduce `try`/`except`: wrap the conversion in a `try` block, catch `ValueError`, and loop back to the prompt. Coverage here is intentionally narrow ‚Äî one exception type, one pattern. The full treatment (raising exceptions, custom types, using exceptions to enforce preconditions) comes in COMP 159.

**CLI:** `grep` as "find lines in a file matching a pattern" ‚Äî existential search at the shell. Students run `grep` on a data file and then write a Python script that does the same job, comparing the two.

### Week 9 ‚Äî More on Lists: Algorithms and Comprehension; IDE Transition

With iteration and lists in place, this week builds the standard list algorithm toolkit and introduces a more concise notation for common patterns. It is also the week students move from Vim to VS Code.

**Programming:** list comprehension as a concise filter-and-transform: `[f(x) for x in lst if p(x)]`. List methods: `append()`, `insert()`, `remove()`, `pop()`, `sort()`, `sorted()`, `reverse()`. Counting, filtering, and searching with loops and comprehensions. String methods: `split()`, `strip()`, `join()`, `lower()`, `replace()`, `find()` ‚Äî strings as sequences of characters.

**Mathematical framing:** a filter is the programming analog of set-builder notation {x ‚àà A | P(x)}. A map (applying a function to every element) is the programming analog of applying f to each element of a sequence. Naming these connections prepares students for the formal set-theory treatment in week 10.

**IDE transition ‚Äî VS Code:** Students have spent eight weeks writing code in Vim. They know a program is a text file; they know the terminal is where it runs. That foundation is exactly why the IDE is introduced now rather than on day one ‚Äî an IDE that hides these layers before students understand them is a liability, not an asset.

Install VS Code and the Python extension. Key affordances to demonstrate:

- *File explorer* ‚Äî the same directory tree students already navigate with `ls` and `cd`, now clickable
- *Syntax highlighting and error squiggles* ‚Äî the editor flags type errors and undefined names before `python3` is ever invoked
- *Integrated terminal* ‚Äî the same shell; `git push`, `make test`, and `python3 script.py` work identically here
- *Debugger basics* ‚Äî set a breakpoint by clicking the line-number gutter; run in debug mode; the debugger pauses and shows the current value of every variable. Step over (execute one line), step into (enter a function call), step out (return from a function). Students use the debugger to trace a list comprehension and a loop ‚Äî the same tracing they have been doing by hand. The debugger does it automatically; the hand-tracing built the mental model that makes the output meaningful.

From this week onward, VS Code is the working environment. Vim is not revisited. The terminal ‚Äî whether accessed directly or through VS Code's integrated terminal ‚Äî remains the submission and testing interface.

**CLI:** `grep`, `sort`, `uniq`, `cut` ‚Äî the shell's own sequence processing tools. Students solve the same problem both ways: a Python script and a one-line shell pipeline. Both approaches are valid; the comparison sharpens the intuition for each.

### Week 10 ‚Äî Sets

Lists are ordered and allow duplicates. Sets are unordered and do not. This week introduces set theory both as mathematics and as a Python data type.

**Mathematical concepts:** sets ‚Äî notation (‚àà, ‚äÑ, ‚äÜ), operations (‚à™, ‚à©, \, complement). Cartesian product: A √ó B is the set of all ordered pairs (a, b) with a ‚àà A, b ‚àà B. Power sets. The distinction between a set and a sequence: {1, 2, 3} = {3, 1, 2}, but [1, 2, 3] ‚âÝ [3, 1, 2].

**Programming:** Python `set` and `frozenset`. Set operations in code: `|`, `&`, `-`, `<=`. Converting between lists and sets. A dict previewed as a structure whose keys form a set ‚Äî lookup by key, not by position.

**File input:** `open()`, iterating over lines, `strip()`, `readline()`. A file is a sequence of lines ‚Äî the same sequence concept as a list, but read from disk one line at a time. The `with open('data.txt') as f:` idiom ensures the file closes after the block. The motivating exercise: read a text file, strip each line, split into words, collect into a set. This is the same task that `sort | uniq` does at the shell; students implement it in Python and compare.

**CLI:** `sort | uniq` produces the set of distinct values from a list. `comm` computes intersection and difference of sorted files. Again, the shell and Python compared side by side.

### Week 11 ‚Äî Number Representation

Where do numbers live in a computer? This week connects the abstract idea of a number to its concrete binary representation, bridging mathematics and the machine.

**Mathematical concepts:** binary (base 2) and hexadecimal (base 16) number systems. Counting and converting between bases. Addition in binary. Why computers use binary: circuits have two stable states; each state represents a bit.

**Programming:** `bin()`, `hex()`, `int('1010', 2)`. Briefly: bitwise operators (`&`, `|`, `^`, `<<`, `>>`). Integers in Python are arbitrary precision ‚Äî they are not limited to 32 or 64 bits ‚Äî but most languages are, so awareness matters.

**CLI:** File permissions in octal (`chmod 644`). `xxd` to view a file's bytes in hexadecimal. Students see that every file ‚Äî including their Python scripts ‚Äî is a sequence of bytes.

### Week 12 ‚Äî Classes and Objects

A class defines a new type: a named collection of values (the instances) with a set of operations (the methods). This week introduces object-oriented programming as a direct extension of the type concept from week 2.

**Mathematical framing:** defining a class is defining a mathematical structure ‚Äî a set of objects together with operations on them. Instance variables are the fields; methods are the operations. The `Vector` class, for example, is a programming version of ‚Ñù‚Åø: its instances are n-tuples, its methods include the dot product operation already implemented in week 7.

**Programming:** `class`, `__init__`, instance variables, `self`, methods, creating instances, dot notation. A first complete class: `Vector`, with a constructor, a `dot()` method that references the work from week 7, and a `__len__` method. Students see that a method is a function attached to a type ‚Äî the type is its domain.

**CLI:** Organizing code into multiple `.py` files. Updating the Makefile for a project with more than one source file.

### Week 13 ‚Äî Designing with Classes

One class is a type. A system has multiple types that interact. This week students design their first multi-class program.

**Mathematical concepts:** relations ‚Äî reflexive, symmetric, transitive, antisymmetric. Equivalence relations and partial orders. These are introduced briefly as the mathematical structures underlying sorted order (`‚â§` is a partial order) and equality testing (equality is an equivalence relation). They will reappear in COMP 271 when discussing sorting algorithms and hash tables.

**Programming:** two classes interacting ‚Äî for example, a `Node` class that holds a value and a `Bag` class that holds a sequence of `Node` instances. Class invariants: a condition that must be true of every well-formed instance. Pre/post conditions on methods. Students write docstrings that state the invariant and the conditions on each method.

## COMP 159

Students entering COMP 159 can navigate the filesystem, write and test Python functions, work with lists and sets, define classes, and use Git and Makefiles. COMP 159 takes these foundations and builds upward: recursion, algorithm analysis, advanced OOP, hashing, and abstract data types.

### Week 1 ‚Äî Recursion: The Concept

Recursion is the technique of solving a problem by reducing it to a smaller version of itself, solving that, and combining the results. It is elegant, it is powerful, and it has a precise mathematical justification: mathematical induction.

**Mathematical framing ‚Äî induction intuition:** to show that a property P holds for all natural numbers n ‚â• 0, show (1) P(0) holds (base case) and (2) if P(n‚àí1) holds, then P(n) holds (inductive step). A recursive function has exactly this structure: the base case handles the smallest input directly; the recursive case assumes the function works on a smaller input and uses that result. Induction is why recursion is correct.

**Programming:** recursive functions in Python. Base case, recursive case. Tracing the call stack for factorial and sum of a list. Common errors: missing base case, recursive step that does not reduce the problem.

---

### Week 2 ‚Äî Recursion Patterns and Recurrence Relations

One worked example is not enough. This week students see multiple recursion patterns and learn to express the cost of a recursive function as a recurrence relation ‚Äî a mathematical equation that describes how the running time on size n depends on the running time on a smaller size.

**Mathematical concepts:** recurrence relations. `T(n) = T(n‚àí1) + c` describes a function that does constant work and one recursive call on a problem one unit smaller ‚Äî its solution is `T(n) = O(n)`. Setting up a recurrence from code: look at the recursive calls and the non-recursive work.

**Programming:** recursive list processing (reverse, flatten, count occurrences). Recursive string processing (palindrome check). The head/tail pattern: operate on the first element, recurse on the rest. Mutual recursion briefly.

**CLI:** `find` as a recursive directory traversal ‚Äî the shell doing recursion on the filesystem.

---

### Week 3 ‚Äî Algorithm Analysis and Big-O

How do we compare algorithms? Two programs can solve the same problem correctly, but one might take seconds and another hours on a large input. Big-O notation gives a mathematical vocabulary for this comparison.

**Mathematical concepts:** Big-O notation ‚Äî formal definition: f(n) = O(g(n)) if there exist constants c and n‚ÇÄ such that f(n) ‚â§ c¬∑g(n) for all n ‚â• n‚ÇÄ. Common classes: O(1), O(log n), O(n), O(n log n), O(n¬≤). Comparing growth rates. Best, worst, and average case.

**Programming:** counting operations in code. Linear search is O(n); looking up an element by index is O(1). The cost of list operations: `append` is O(1) amortized, insertion at position 0 is O(n). Algorithm choice matters.

**CLI:** the `time` command. Benchmarking from a shell loop: `for n in 100 1000 10000; do time python3 sort.py $n; done`. Students observe growth empirically before analyzing it mathematically.

---

### Week 4 ‚Äî Sorting

Sorting is the canonical algorithm design problem. This week covers quadratic sorting (simple, instructive) and then merge sort (recursive, efficient, and the payoff of recurrences).

**Algorithms:** selection sort and insertion sort ‚Äî both O(n¬≤), both easy to implement, good for tracing by hand. Merge sort: divide the list in half, sort each half recursively, merge the results. Runtime: `T(n) = 2T(n/2) + n`. Solving by unrolling: `T(n) = 2(2T(n/4) + n/2) + n = 4T(n/4) + 2n = ... = O(n log n)`. This is the first non-trivial recurrence solution and the payoff of weeks 2 and 3 together.

**Mathematical tools:** the Master Theorem as a lookup recipe (not derived) for recurrences of the form `T(n) = aT(n/b) + f(n)`.

**CLI:** `sort` with options (`-n`, `-r`, `-k`). Running shell sort and Python sort on the same file and comparing.

### Week 5 ‚Äî Inheritance

Inheritance allows a new class to extend an existing one, adding or modifying behavior without rewriting what already works. It is one of the central mechanisms of object-oriented design.

**Mathematical framing:** subclass as a subset. If `Dog` is a subclass of `Animal`, then every `Dog` is an `Animal` ‚Äî `Dog ‚äÜ Animal` in the set sense. The subclass inherits all operations of the superclass and may add more. Liskov substitution principle, informally: wherever an `Animal` is expected, a `Dog` may be used.

**Programming:** `class Dog(Animal)`, method inheritance, `super()`, method overriding, `isinstance()`. A worked example with a clear hierarchy: `Shape` ‚Üí `Circle`, `Rectangle`, `Triangle`. Each subclass overrides `.area()` and `.perimeter()`.

**CLI:** organizing a multi-file project with an inheritance hierarchy. Updating the Makefile.

### Week 6 ‚Äî Polymorphism and Multiple Interacting Classes

Polymorphism means that the same method name resolves to different behavior depending on the type of the object. Multiple interacting classes means that a working system is not one class but several, each with a clear responsibility.

**Programming:** polymorphism in action ‚Äî a list of `Shape` objects, a loop that calls `.area()` on each, each call dispatching to the correct subclass implementation. Designing multi-class systems: `Node` holds a value and a reference to another `Node`; a `LinkedContainer` holds a sequence of `Node` objects via a chain of references. This is the pattern COMP 271 will use at speed ‚Äî students should encounter it here first, at a slower pace, before they need it under pressure.

**CLI:** running each class independently using a `__main__` block; integration testing by running the full program. `diff`-based output testing in the Makefile.

### Week 7 ‚Äî Dictionaries and Hashing

A dict maps keys to values. It is the Python implementation of an associative array ‚Äî a function from keys to values, implemented using hashing.

**Mathematical concepts:** a hash function as a function from keys to bucket indices ‚Äî same domain/codomain vocabulary as COMP 158 week 4. The pigeonhole principle in one paragraph: if there are more possible keys than buckets, at least two keys must land in the same bucket. This is inevitable; collision handling is the engineering response. Not a full session on the pigeonhole principle ‚Äî a paragraph connecting it to the data structure.

**Programming:** Python `dict`. Creation (`{}`), lookup, insertion, `in`, `.items()`, `.keys()`, `.values()`, iteration, dict comprehension. Use cases: counting word frequencies, grouping items by a property, memoizing recursive functions (computing Fibonacci with memoization as a worked example).

**CLI:** word frequency analysis ‚Äî reading a file, counting with a dict, writing sorted results. Then doing the same with `sort | uniq -c | sort -rn`. Both approaches, compared.

### Week 8 ‚Äî Mutability and Object References

In Python, assignment does not copy values ‚Äî it copies references. Two names can refer to the same object. Mutating the object through one name changes what the other name sees. This is confusing until it is understood; once understood, it is the key to reasoning about linked structures.

**Programming:** mutable vs. immutable types. Assignment copies a reference, not a value. Aliasing: `a = b; a.append(1)` changes what `b` sees. Shallow copy vs. deep copy. Why this matters for linked structures: a `Node` that stores a reference to another `Node` is not storing a copy of that node ‚Äî it is storing a pointer to it. Mutating the second node through one reference is visible through all references to it.

**CLI:** demonstrating aliasing interactively at the Python REPL in the terminal. `id()` to show that two names refer to the same memory location.

### Week 9 ‚Äî Probability and Expected-Case Analysis

Worst-case analysis with Big-O is conservative. Expected-case analysis asks: given a random input, what is the typical running time? This requires probability.

**Mathematical concepts:** sample spaces, events, basic probability axioms. Expected value: E[X] = ‚àë x ¬∑ P(X = x). Expected number of comparisons in a linear search when the target is equally likely to be at any position: (n+1)/2. Expected collision behavior in a hash table.

**Programming:** simulation using the `random` module. Estimating œÄ by Monte Carlo. Simulating a hash table with random keys and a fixed number of buckets ‚Äî counting collisions, comparing observed rates to predicted rates. Memoized Fibonacci from week 7 revisited: how does memoization change the call structure?

**CLI:** shell loops to run repeated simulations and collect results into a file for analysis.

### Week 10 ‚Äî Abstract Data Types

An abstract data type specifies *what* a collection does ‚Äî its operations and their contracts ‚Äî without specifying *how* it is implemented. The same ADT can have multiple implementations with different performance characteristics.

**Programming:** stack (LIFO) and queue (FIFO). Implement each with a Python list. Write a full test suite with `assert`. Then implement the stack a second time using a linked chain of `Node` objects (from week 6). Run the same tests ‚Äî they pass, because the interface is the same. The point: the ADT is the contract; the implementation is interchangeable.

**Mathematical framing:** the ADT interface is like a function signature ‚Äî it specifies the domain and codomain of each operation without exposing the internal structure. Encapsulation as information hiding.

**CLI:** Makefile managing a multi-module project. Each ADT in its own `.py` file. `make test` runs all suites.

### Week 11 ‚Äî Shell Scripting in Depth

Shell scripts orchestrate programs. A non-trivial project involves multiple steps ‚Äî compile, test, clean ‚Äî and a shell script or Makefile automates them. This week turns scripting from a convenience into a skill.

**CLI:** Shell functions: defining and calling. Positional parameters: `$1`, `$2`, `$@`, `$#`. Branching: `if`/`elif`/`else` in bash. Loops over files: `for f in *.py; do python3 "$f"; done`. A complete script: takes a directory argument, finds all `.py` files, runs each through a style checker, reports failures.

**Programming:** designing Python programs to compose with shell tools. `sys.argv` for receiving arguments from the shell. Reading from `stdin`, writing to `stdout`. Using exit codes to signal success or failure. A Python program and a shell script together as a pipeline.

**Mathematical note:** a shell pipeline is function composition. If `f` maps input to intermediate output and `g` maps intermediate to final output, then `f | g` is `g ‚àò f`. Students have been using this since COMP 158; now it has a name.

### Week 12 ‚Äî Testing and Code Quality in Depth

A program that works on the examples you tried is not the same as a program that works correctly. This week focuses on systematic testing and on the discipline of writing code that communicates its intent.

**Programming:** writing complete test suites. Testing edge cases: empty input, single element, duplicate elements, maximum size. `diff`-based output testing: run the program, compare its output to a `.expected` file using `diff`. If `diff` produces no output, the test passes. Docstrings in depth: `Args`, `Returns`, `Raises`. `help()` and `__doc__`. Brief peer review: read a classmate's docstrings and tests before reading their implementation.

**Exception handling ‚Äî full treatment:** When a precondition is violated, raising an exception is the correct response: `raise ValueError("expected a non-empty list")`. This is what the `Raises` field in a docstring documents. Students write methods that raise on bad input, write tests that verify the exception is raised (`try`/`except` in a test function, checking that the right exception type is caught), and see how this connects to the pre/post condition framework ‚Äî a violated precondition is not a bug in the function, it is a contract breach by the caller.

**Mathematical connection:** loop invariants ‚Äî a predicate that holds before the loop, is maintained by each iteration, and therefore holds after the loop. Pre/post conditions expressed with ‚àÄ: "for all inputs satisfying the precondition, after the function returns, the post-condition holds."

**CLI:** `make test` with `diff` checking. Committing only after all tests pass. Running the full suite in CI: a brief demonstration that the same `make test` command works identically on a different machine.

### Week 13 ‚Äî Language-Agnostic Thinking

Students who complete COMP 159 will take COMP 271 in Python and then COMP 272 in Java. The syntactic transition to Java should not be a conceptual barrier. This week reviews every major programming concept through a language-neutral lens.

**Concepts reviewed without Python syntax:** iteration (stepping through a collection, not `for` loops), recursion (base case + inductive step, not Python's call stack), OOP (state + behavior + encapsulation, not `class` and `self`), type systems (a type is a set of values with operations, not `int` or `str`), testing (specifying and verifying a contract, not `assert`).

### Week 14 ‚Äî Capstone

Full project: a multi-class Python program that must include inheritance, at least one recursive operation, a dict for some lookup or grouping, file input, and a complete test suite. Submitted via `git push`. Built and tested with `make`. Fully documented with docstrings.

The oral exit interview (from the assessment framework) is conducted this week. Students should be able to sit at any machine, open VS Code, navigate the filesystem, run `make test` from the integrated terminal, and explain what each class does and why. The interview also asks students to demonstrate that they can navigate to their project from a plain terminal without the IDE ‚Äî the filesystem skills from week 1 have not been replaced, only augmented.

Review: every mathematical concept from the two semesters mapped to the programming concept it underlies. Students leave with a vocabulary that works in both directions.
